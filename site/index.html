mcp-mirror

Your MCP server's tool is not the tool the LLM sees.

Between your MCP server and the model sits a layer you didn't write: your agent framework's adapter. mcp-mirror runs your real MCP server through five real framework adapters at once and shows you, side by side, exactly what each one hands to the LLM.

Patterns across all 100 tools

Pydantic AI

+1 everywhere

Returns the raw mcp.types.Tool. Adds only an internal class-name marker. The most faithful adapter we measured — by a long way.

LangChain

-1 +1 everywhere

Drops the protocol-level _meta, adds an LC tool-class marker. Eerily consistent. Input schema preserved byte-for-byte.

AG2

-3 to -4 +1

Like LangChain but drops outputSchema, icons, annotations on top of _meta.

LlamaIndex

Highly variable

Converts JSON Schema → Pydantic → JSON Schema. Simple tools drift mildly. Complex tools restructure heavily — $defs blocks, reorganized properties, dozens of new fields.

CrewAI

Most invasive

Extends descriptions 5–6× with usage-guidance text, restructures schemas through mcpadapt + CrewAI Pydantic generation. 62 deltas on Linear_CreateIssue alone.

How mcp-mirror gets these numbers

  1. Real MCP server. Either the bundled reference server (rich-schema fixtures, runs via stdio) or a real Arcade gateway (streamable-HTTP, OAuth-protected). No mock servers, ever.
  2. Spec-correct authentication. For Arcade gateways, mcp-mirror discovers the OAuth metadata from the WWW-Authenticate challenge, registers itself via Dynamic Client Registration, runs PKCE with the gateway URL as the RFC 8707 resource indicator, and caches the token. Real MCP-spec auth, exactly how a production agent would do it.
  3. Five real framework adapters. Each capture loads the same MCP server through the framework's actual published integration — langchain-mcp-adapters, llama-index-tools-mcp, crewai-tools, pydantic_ai.mcp, autogen.mcp. No re-implementation, no simulation.
  4. Introspect at the adapter's output. mcp-mirror reads what each framework exposes to its agent: tool.args_schema for LangChain, tool.metadata.fn_schema for LlamaIndex, tool.inputSchema for Pydantic AI, etc. That's the same shape the LLM eventually sees.
  5. Structural diff against the server announcement. A recursive JSON-Schema diff classifies every field-level delta as faithful, additive, lossy, or transformative. The overall category for a tool is the worst delta it contains.

Reproducing this locally

git clone https://github.com/thierrypdamiba/mcp-mirror
cd mcp-mirror
python3.13 -m venv .venv
.venv/bin/pip install -e '.[all]'

# .env (1Password references resolved at runtime by `op run`)
cat > .env <<'EOF'
OPENAI_API_KEY=op://Engineering - Shared/Engine Local Dev/AI/OPENAI_API_KEY
ARCADE_API_KEY=op://Private/td arcade/credential
ARCADE_USER_ID=thierry@arcade.dev
ARCADE_GATEWAY_URL=https://api.arcade.dev/mcp/mcp-mirror
EOF

# First run: opens a browser once, caches the token under ~/.cache/mcp-mirror/
op run --env-file=.env -- mcp-mirror --arcade

What this means for tool authors

The diff isn't about which framework is "right" — every framework's transformation is internally coherent for its own runtime. The diff is about what your tool actually looks like when the LLM picks it up, which is the only thing that matters for tool-selection accuracy.

Three implications worth carrying back to your team:

  • Test your tool across frameworks, not just one. A schema that's faithful in Pydantic AI may be unrecognizable to a CrewAI agent — same tool, different downstream behavior.
  • Defensive design patterns. Keep tool descriptions tight enough that 5× growth still fits in context. Keep schemas simple enough that $defs regeneration stays sane.
  • This is a measurable layer. Run mcp-mirror in CI against the framework versions you ship, and you'll catch breakage at the adapter layer before your users do.
